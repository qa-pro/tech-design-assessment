## 1. 评分框架总览

本标准采用**四维综合评分机制**,总分100分,从产品、后端、前端、测试四个专业视角全方位评估后端技术方案质量。

| 维度 | 权重 | 核心考核点 |
| :--- | :--- | :--- |
| **产品视角** | **30%** | 目标与成果、场景与流程、服务承诺、风险与约束、系统依赖、数据合规、验收观测、发布运营 |
| **后端视角** | **30%** | 需求范围、架构设计、数据存储、接口集成、性能容量、可观测性、安全合规、运维发布、测试质量、代码协作、规划迭代 |
| **前端视角** | **20%** | 场景旅程、API契约、鉴权权限、性能指标、联调流程、发布灰度、观测告警、数据一致性、协作文档 |
| **测试视角** | **20%** | 业务场景、异常兜底、数据接口、性能容量、安全合规、观测故障、发布回滚、变更度量 |

## 2. 评分细则

### 2.1 产品视角 (30分)

#### 2.1.1 目标与成果 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **业务痛点复述** | 1.5 | 方案复述要解决的业务痛点、成功指标(北极星或阶段KPI),与PRD完全对齐。 |
| **交付物映射** | 1.5 | 所有交付物(接口、消息、批任务、配置等)可映射到明确的产品场景或用户旅程。 |
| **现网影响说明** | 1 | 对现网的影响(新增/替换/废弃能力)以及与旧流程的切换策略已写明。 |
| **价值验证路径** | 1 | 说明如何验证方案是否解决了业务痛点,包含可度量的验证方式。 |

#### 2.1.2 场景与流程 (5分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **端到端流程图** | 2 | 核心流程、时序/泳道图能够覆盖所有入口、触发条件、异常分支,且产品看得懂。 |
| **多端多角色覆盖** | 1.5 | 多端/多角色(C端、B端、运营、客服等)的体验承诺在方案中逐一落地。 |
| **异常补救策略** | 1 | 异常或失败路径的业务补救方式(兜底、告警、人工处理)与产品体验一致。 |
| **场景完整性** | 0.5 | 所有业务场景、触发点、边界条件被完整枚举,无遗漏。 |

#### 2.1.3 服务承诺与指标 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **SLA/QoS定义** | 1.5 | 技术方案中列出的SLA/QoS(QPS、延迟、吞吐、幂等要求)已得到产品认可,并满足对外承诺。 |
| **容量测算依据** | 1 | 方案包含容量测算依据或引用的业务预测,产品确认假设来源可靠。 |
| **运营排期对齐** | 1 | 对外或对内宣称的运营排期、灰度窗口、客户影响面与方案里程碑保持同步。 |
| **性能基线说明** | 0.5 | 提供性能基线数据或对比参考,说明性能目标设定的合理性。 |

#### 2.1.4 风险与约束 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **前提条件清单** | 1.5 | 关键前提条件(数据可得性、外部系统改造、合规批复等)在方案中被显式列出并有责任人。 |
| **回滚应急路径** | 1 | 方案提供回滚/应急路径,能满足产品在事故时的体验底线。 |
| **高风险项识别** | 1 | 已识别的高风险项(技术、资源、依赖)给出了缓解或替代方案,且产品理解其业务影响。 |
| **约束条件说明** | 0.5 | 明确列出技术、业务、合规等方面的约束条件,说明不可突破的限制。 |

#### 2.1.5 系统依赖与协同 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **依赖系统一览** | 1.5 | 所有内部/外部依赖系统一览完善:接口名称、联系人、SLA、对接状态、Lead Time。 |
| **调用方式明确** | 1 | 调用方式(同步/异步/批)、鉴权方式、密钥管理责任划分清晰,满足安全要求。 |
| **交付排期对齐** | 1 | 依赖的交付排期与本项目关键里程碑对齐,存在冲突时已有缓冲或替代方案。 |
| **依赖风险评估** | 0.5 | 对关键依赖进行风险评估,说明依赖失败时的降级或替代方案。 |

#### 2.1.6 数据、权限与合规 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **数据流转路径** | 1.5 | 数据流转路径(读/写源、字段粒度、主数据归属)被完整绘制,满足数据主责要求。 |
| **敏感数据处理** | 1 | 敏感字段、脱敏策略、访问最小化、留存周期等合规点已在方案中落实,并贴合产品承诺。 |
| **权限模型说明** | 1 | 权限模型(角色/租户/地域)对业务体验的影响被说明,如有限制则写明对策。 |
| **合规要求映射** | 0.5 | 数据主权、留存周期、跨境/跨租户限制等合规要求在方案中被映射到具体实现。 |

#### 2.1.7 验收与观测 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **测试用例清单** | 1 | 方案提供可执行的核心/异常/回归测试用例清单,并与产品验收标准一一对应。 |
| **观测指标定义** | 1 | 日志、埋点、指标、报表等观测项写明采集方式与判责口径,以支撑产品上线验收。 |

#### 2.1.8 发布、运营与沟通 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **灰度策略说明** | 1 | 灰度策略(批次、比例、地域/租户)明确,与产品对客户/运营的沟通节奏一致。 |
| **运营支持点** | 1 | 配置项、开关、客服/运营后台操作点在方案中列出,并有培训/文档计划。 |

### 2.2 后端视角 (30分)

#### 2.2.1 需求与范围对齐 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **业务场景同步** | 1 | 业务场景、输入条件、异常路径与PRD完全同步,无遗漏的触发点。 |
| **系统职责定义** | 1 | 系统职责、写入/消费边界、幂等/一致性要求被明确定义。 |
| **待确认事项** | 1 | 方案列出所有待确认的业务前提,并注明依赖方、截止时间。 |

#### 2.2.2 架构与流程设计 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **架构图完整性** | 1.5 | 提供端到端架构图/流程图/状态机,能够推演主要链路与异常分支。 |
| **模块职责划分** | 1 | 模块职责划分符合现有架构原则,复用策略和不复用原因写清。 |
| **核心逻辑说明** | 1 | 关键算法、规则引擎、任务调度等核心逻辑有足够说明,避免黑盒。 |
| **扩展性设计** | 0.5 | 说明架构的扩展性设计,如何应对未来需求变化。 |

#### 2.2.3 数据模型与存储 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **数据实体设计** | 1.5 | 新增/变更的数据实体、表结构、索引、分区策略、热点字段均有设计说明。 |
| **一致性策略** | 1 | 数据一致性策略(强一致、最终一致、补偿)与业务SLA匹配。 |
| **数据迁移方案** | 1 | 历史数据回填/迁移方案、双写策略、回滚流程被定义并评估可行性。 |
| **存储选型说明** | 0.5 | 存储选型(关系型/NoSQL/缓存等)有明确依据和对比分析。 |

#### 2.2.4 接口与集成 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **接口契约定义** | 1.5 | 枚举所有对外/对内接口、消息、批任务,含契约、字段、错误码。 |
| **调用策略明确** | 1 | 调用方式(同步/异步)、重试/超时/限流策略明确,互斥条件可追踪。 |
| **外部依赖确认** | 1 | 外部依赖(第三方、兄弟团队)的交付排期、SLA、联调窗口已确认。 |
| **版本兼容性** | 0.5 | 接口版本策略、向后兼容方案、灰度发布规则清晰。 |

#### 2.2.5 性能与容量 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **容量测算依据** | 1 | 提供峰值/平均QPS、吞吐、数据体量、作业窗口等测算依据。 |
| **瓶颈分析** | 1 | 瓶颈分析(CPU、IO、网络、锁、DB资源)和扩展策略(分片、缓存、异步化)明确。 |
| **压测计划** | 1 | 压测计划、目标指标、执行责任人和回归策略写入方案。 |

#### 2.2.6 可观测性 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **日志埋点定义** | 1.5 | 关键链路的日志、埋点、Trace、Metric指标定义完整,可落地到现有观测平台。 |
| **告警策略** | 1 | 告警策略(阈值、抖动、值班人)与业务影响等级对应,避免盲点或告警风暴。 |
| **故障排查支持** | 0.5 | 故障排查所需的关联ID、上下文信息在设计阶段已考虑。 |

#### 2.2.7 安全与合规 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **认证授权设计** | 1 | 认证、授权、权限校验点与现有体系兼容,异常路径能安全降级。 |
| **敏感数据保护** | 1 | 敏感数据的传输/存储加密、脱敏、访问审计方案已有实现路径。 |
| **合规要求落实** | 1 | 数据主权、留存周期、跨境/跨租户限制等合规要求在方案中被映射。 |

#### 2.2.8 运维与发布 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **配置管理** | 1 | 配置项、灰度开关、Feature Flag及默认值说明清楚,支持可控发布。 |
| **发布回滚策略** | 1 | 发布/回滚策略(蓝绿、金丝雀、紧急回滚)可操作,且覆盖依赖服务。 |
| **运行手册** | 1 | 运行手册包含常见故障、应急预案、人工补救步骤与通知链路。 |

#### 2.2.9 测试与质量保障 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **测试策略** | 1 | 单元、集成、契约、回归、性能等测试策略明确,自动化覆盖目标量化。 |
| **Mock仿真方案** | 1 | Mock/仿真/沙箱方案完善,联调前即可验证关键路径。 |

### 2.3 前端视角 (20分)

#### 2.3.1 场景与用户旅程 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **关键路径支持** | 1.5 | 关键用户路径(含多端、多角色)在后端方案中有对应的接口/事件支持,未遗漏。 |
| **多场景切换** | 1 | 多场景切换(登录态、租户、地域、AB实验)对前端体验的影响有明确描述。 |
| **异常兜底对齐** | 1 | 异常/兜底体验(错误提示、重试、人工处理)前后端对齐。 |
| **用户旅程完整性** | 0.5 | 用户旅程的每个步骤都有对应的后端支持,无断点。 |

#### 2.3.2 API契约与数据模型 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **接口列表完整** | 1.5 | 所有接口列表、字段结构、类型、可空性、默认值、分页/排序规则明确。 |
| **版本策略清晰** | 1 | 版本策略(path、header、参数)、向后兼容方案、灰度发布规则清晰,避免强制升级。 |
| **错误码体系** | 1 | 错误码体系、文案、可定位信息(requestId、traceId)满足前端展示与排障需求。 |
| **数据模型文档** | 0.5 | 提供完整的数据模型文档,包含字段说明、枚举值、示例数据。 |

#### 2.3.3 鉴权与权限 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **登录态管理** | 1 | 登录态、Token、权限/角色校验与前端逻辑保持一致,涉及重定向或刷新时序已说明。 |
| **跨端状态共享** | 1 | 跨端共享状态(Web/App、小程序、桌面)所需的Header、cookie、存储策略明确。 |
| **敏感操作校验** | 1 | 敏感操作是否需要二次校验(验证码、OTP、二次确认)在后端方案中给出交互信号。 |

#### 2.3.4 性能指标与约束 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **接口SLA定义** | 1 | 接口SLA(延迟、吞吐)、数据体量、分页上限与前端性能预算匹配。 |
| **批量查询优化** | 1 | 批量查询/长列表场景是否提供增量、压缩、流式、缓存等手段,避免前端重复拼装。 |
| **实时性策略** | 1 | 实时性要求、轮询/推送策略、消息节流对前端实现的影响已评估。 |

#### 2.3.5 联调流程 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **Mock产出计划** | 1 | Mock、Swagger、Schema、示例数据产出计划明确,具备可用性里程碑。 |
| **环境配置对齐** | 1 | 沙箱/预发环境的账号、数据、Feature Flag、依赖开关与前端对齐。 |

#### 2.3.6 发布与灰度协同 (2分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **灰度计划匹配** | 1 | 后端灰度计划与前端发布节奏匹配;多版本共存期间的兼容策略已定义。 |
| **配置透明化** | 1 | 配置、开关、实验参数的生效顺序和回滚策略对前端透明。 |

#### 2.3.7 观测与告警 (1分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **业务指标监控** | 0.5 | 关键业务指标(成功率、首屏数据完整度、关键按钮可用性)在后端方案中有监控点。 |
| **降级策略约定** | 0.5 | 告警触发后是否有前端可见的降级策略(占位、默认值、提示)已约定。 |

#### 2.3.8 数据一致性与离线需求 (1分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **数据刷新策略** | 0.5 | 数据刷新频率、缓存失效策略、最终一致性窗口满足前端展示预期。 |
| **离线场景支持** | 0.5 | 导出、报表、离线文件生成等场景的状态回传或轮询接口定义齐全。 |

### 2.4 测试视角 (20分)

#### 2.4.1 业务场景与验收口径 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **场景映射完整** | 1.5 | 技术方案完整映射PRD场景,QA能基于方案输出用例矩阵。 |
| **输入前置条件** | 1 | 每个场景的输入、前置条件、鉴权/权限要求明晰,可生成测试数据。 |
| **判定口径明确** | 1 | 成功/失败判定、用户可见效果、日志/指标验证口径在方案中明确。 |
| **用例可执行性** | 0.5 | 测试用例具备可执行性,包含明确的测试步骤和预期结果。 |

#### 2.4.2 异常与兜底策略 (4分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **异常链路描述** | 1.5 | 异常链路(依赖失败、超时、限流、数据缺失)在方案中逐一描述,可设计对应用例。 |
| **兜底补偿机制** | 1.5 | 兜底/补偿/重试机制的触发条件、次数、幂等要求可测试。 |
| **回滚修复方式** | 1 | 回滚、手工修复、数据校正方式写明,QA能验证其有效性。 |

#### 2.4.3 数据与接口契约 (3分)

| 考核项 | 分值 | 评分标准 |
| :--- | :--- | :--- |
| **数据模型完整** | 1 | 数据模型、字段类型、枚举值、可空性、默认值等信息足够支撑测试设计。 |
| **API契约清晰** | 1 | API契约提供错误码、文案、幂等键、分页/排序规则,便于验证边界。 |
| **数据迁移影响** | 1 | 历史数据迁移/回填/双写对测试环境的影响、所需脚本、校验口径清晰。 |

#### 2.4.4 性能与容量验证 (3分)

| **压测场景明确** | 1 | 压测场景(热点接口、批任务、链路长流程)和目标指标明确,含达标判定方式。 |

## 3. 一票否决项

如果出现以下情况,评审直接不通过:

1. **缺少端到端架构图/流程图/状态机,无法推演主要链路。**
2. **缺少接口契约定义,未枚举所有对外/对内接口、消息、批任务。**
3. **缺少数据模型设计,未说明新增/变更的数据实体、表结构、索引。**
4. **缺少系统依赖清单,未列出所有内部/外部依赖系统及SLA。**
5. **缺少性能容量测算,未提供QPS、吞吐、数据体量等依据。**
6. **缺少安全合规设计,未说明认证授权、敏感数据保护方案。**
7. **缺少发布回滚策略,未定义灰度方案和回滚路径。**
8. **缺少测试策略,未说明单元、集成、回归、性能测试计划。**
9. **缺少关键用户路径的后端支持,存在前端无法实现的场景。**
10. **缺少错误码体系,前端无法进行错误展示和排障。**
11. **缺少异常链路描述,无法设计异常测试用例。**
12. **缺少数据迁移方案,未说明历史数据回填/双写策略。**
13. **缺少压测计划,未定义压测场景和目标指标。**
14. **缺少观测指标定义,未说明日志、埋点、Trace、Metric。**
15. **缺少验收标准,未明确成功/失败判定口径。**

## 5. 评分说明

### 5.1 评分原则

1. **证据导向**:每项打分严禁凭空判定,必须在技术方案中找到对应依据。
2. **具体引用**:
   - **必须引用**:具体章节号、页码、表格标题或附件名称。
   - **必须摘录**:复制原文关键句或对逻辑进行转述(Paraphrase)。
3. **问题闭环**:对于扣分项,需明确指出缺失内容或逻辑冲突点,并提出具体补充期望。
4. **冲突处理**:若发现跨章节冲突,需在记录中明确两个出处并说明影响。

### 5.2 评分等级

- **优秀(90-100分)**:方案完整、清晰、可执行,各视角需求充分满足。
- **良好(80-89分)**:方案基本完整,存在少量非关键问题,不影响开发。
- **合格(60-79分)**:方案存在部分缺失或模糊,需要补充后才能进入开发。
- **不合格(<60分)**:方案存在严重缺失或一票否决项,需重新编写。
